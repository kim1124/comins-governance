# Comins Contract v1.1 Adoption Completion Design

## Goal

Complete the locally prepared Comins Contract v1.1 adoption without coupling the independent repositories or performing remote writes. The result must leave governance as the shared policy source while each module keeps its own CI, release workflow, verification, and commit history.

## Scope

- Finalize the governance v1.1 contract, changelog, module checklist, security policy, release policy, and module AGENTS template.
- Adopt the contract and model policy in `comins-table`, `comins-grid-layout`, and `comins-sortable` through separate repository commits.
- Add the missing `comins-grid-layout` verification workflow and changelog.
- Document supported React and browser ranges for `comins-table` and `comins-grid-layout`.
- Add a Grid Layout consumer smoke check and package-local, manually dispatched npm trusted-publishing workflows for the two implemented packages.
- Verify remote repository metadata and report security settings that cannot be read with the available authenticated tools.

Remote pushes, GitHub setting mutations, npm publication, tags, and GitHub Releases are out of scope.

## Repository Boundaries

`comins-governance` owns shared policy only. Each package repository records its adopted contract version and implements the policy through package-local files. No workspace, source synchronization, linked dependency, or shared release workflow is introduced.

Commits remain repository-local:

1. Governance design and Contract v1.1 policy.
2. Comins Table adoption and release-readiness configuration.
3. Comins Grid Layout standalone documentation, adoption, CI, and release-readiness configuration without mixing unrelated package work.
4. Comins Sortable policy adoption only; package scaffolding remains deferred.

## CI And Release Design

The Grid Layout verification workflow runs immutable installation and the existing full verification gate on pull requests and pushes to `main`.

Neither package exists on the npm registry as of 2026-07-15. npm requires a package to exist before staged publishing or trusted-publisher registration. The first publication is therefore a one-time interactive maintainer bootstrap using npm account 2FA. Immediately after that publication, the maintainer registers the repository workflow as the trusted publisher, restricts it to `npm stage publish`, and disallows token publishing.

Each implemented package receives a post-bootstrap manual trusted-publishing workflow with these guards:

- `workflow_dispatch` only; no automatic tag or Release trigger.
- A required version input that must equal `package.json`.
- Node 24 on a GitHub-hosted runner, npm 11.15.0, disabled release-build package caching, and immutable installation.
- Full package verification, consumer smoke testing, and `npm pack --dry-run` before staging.
- GitHub `npm` environment usage so repository-side approval can be configured independently.
- Job-level `id-token: write` and read-only repository contents.
- `npm stage publish --access public`; no long-lived npm token in repository configuration.

The workflow file prepares the repository side only. Initial interactive publication, trusted-publisher registration, stage-only permission, environment reviewers, maintainer 2FA, dependency alerts, secret scanning, and push protection remain GitHub or npm account settings and must be verified separately before publication. A maintainer approves or rejects the staged version with 2FA outside CI.

## Documentation Contract

The two implemented packages explicitly document:

- supported React and React DOM peer ranges;
- current stable Chromium-based Chrome and Edge support, with the Playwright-bundled Chromium as the continuously verified browser; Firefox and Safari remain outside the supported contract until their browser projects are added;
- browser-only or client-boundary behavior and current SSR limitations;
- release status and changelog expectations.

Sortable remains a planned module. It adopts governance and security guidance but does not gain speculative package metadata, CI, or release configuration before its API and runtime dependency are approved.

## Failure Handling

- Treat workflow syntax, immutable-install, test, build, package-content, and consumer-smoke failures as blocking.
- Refuse to run the OIDC workflow before the package exists and its exact repository, workflow filename, and `npm` environment are registered on npmjs.com.
- Keep staged and unstaged Grid Layout changes visible and commit only reviewed paths.
- Do not claim remote security controls are enabled unless an authenticated API or maintainer evidence confirms them.
- Do not push, publish, tag, or mutate GitHub settings during this work.

## Verification

- Governance: reference checks, contract/template version consistency, placeholder scan, and `git diff --check`.
- Comins Table: workflow syntax inspection, `npm run verify`, focused documentation tests, consumer smoke test, package dry run, and `git diff --check`.
- Comins Grid Layout: workflow syntax inspection, `npm run verify:full`, package dry run, and staged/unstaged diff checks.
- Comins Sortable: Markdown link/reference inspection and `git diff --check`.
- All repositories: confirm intended commit scope and leave branches unpushed.
